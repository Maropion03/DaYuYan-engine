# 答于言 · 技术架构说明

## 一、核心设计原则

| 原则 | 实现 |
|------|------|
| **数据不出端** | 所有推理在本机 Snapdragon NPU 完成，零云调用 |
| **多模态融合** | Qwen2.5-VL-3B 视觉 + Tesseract 文字 + PyMuPDF 文本层互补 |
| **场景化判断** | 4 类文档各自走专用 prompt + 后处理管线，非通用 OCR |
| **小模型稳定可控** | 3B 参数 + 2048 上下文极限下，通过 prompt 工程 + 多层修正达成生产可用 |

## 二、四层服务架构

### 2.1 端口分布

| 端口 | 服务 | 进程 | 职责 |
|------|------|------|------|
| 8000 | 前端静态服务 | python (serve_snapextract.py) | 提供 HTML / JS / 演示样例下载 |
| 8765 | 多模态代理 | python (proxy.py) | `/chat` 转 Qwen · `/ocr` 调 Tesseract · `/pdf` 抽 PDF 文本层 |
| 8766 | 场景分析后端 | python + FastAPI (scene_runtime/app.py) | 4 场景深度判断（推荐/风险/匹配等） |
| 8910 | Qwen NPU 推理 | GenieAPIService.exe | Qwen2.5-VL-3B 多模态推理（NPU） |

### 2.2 调用链

**文档解析阶段：**

```
浏览器 → :8000 (页面加载) → :8765/ocr (Tesseract OCR) ─┐
                          → :8765/pdf (PyMuPDF 抽取) ─┼─→ 合并 OCR 文本
                          → :8765/chat → :8910 (Qwen 视觉结构化) ┘
                                                       │
                                              结构化字段 + 表格 + 公式
```

**场景分析阶段：**

```
浏览器 → :8766/api/scene-analysis/run
       │
       │ 后端内部读文件 → 提取 OCR → 走专用 prompt
       │
       ▼
   返回判断结果（评分、风险、建议、面试问题等）
```

## 三、Qwen2.5-VL-3B 关键约束与应对

### 3.1 硬约束：上下文 2048 token

模型编译产物路径中的 `ar128_cl2048` 明确写死了上下文长度。

**应对：**
- 结构化 prompt 压到 ≤700 token
- OCR 摘录截断到 600 字符
- 图像 token（256-1024）+ 输出 token（≤620）必须共同满足 < 2048
- 修复重试 prompt 中失败响应截断到 800 字符

### 3.2 软约束：3B 模型易陷入复读循环

**应对：**
- `presence_penalty: 1.2` + `frequency_penalty: 0.8`
- stop sequences: `['\n```', '\n\n\n\n\n\n']`
- 单元素数组 / 重复字段强制去重
- 输出严格单行 JSON

### 3.3 软约束：占位符回写

3B 模型容易把 schema 中的 `""` 当作示例文本回写为 `[Title]` / `[Party A]` 等占位符。

**应对：**
- prompt 中加 STRICT 规则禁止
- 前端 `normalizeStructuredParse()` 内置 placeholder 检测正则，识别后剥离
- 论文场景额外用 OCR 文本层重写 title / authors

## 四、4 场景专用管线

### 4.1 论文（paper）

```
1. Qwen 视觉抽取 (per page) → {title, authors, venue, abstract, ...}
2. PyMuPDF 文本层 → refinePaperFieldsFromOcr() 重写 title/authors
3. Qwen 多模态补充 → 公式 LaTeX + 图表说明
4. Tesseract 文本 → countFromOcrText() 数 Eq.N / Figure N 编号引用
5. 用户输入"研究方向" → :8766 场景分析 → 给出 deep_read/skim/skip 判断
```

### 4.2 简历（resume）

```
1. Qwen 视觉抽取 → 教育/工作/技能等结构化字段
2. buildResumeParseFromOcrText() 从 OCR 文本层补全 contact + 项目经历
3. 用户贴 JD → :8766 → 输出 三维评分（技能/经验/匹配）+ 面试问题 3-5 条
```

### 4.3 合同（contract）

```
1. Qwen 视觉抽取 → 主合同结构化（甲乙方/金额/条款）
2. extractContractFieldsFromOcr() 用 OCR 标签提取兜底
3. 用户上传修改版 → :8766 → 5 处变更对比 + 风险评级 + 履约里程碑
```

### 4.4 对账（statement）

```
1. Qwen 视觉抽取 → 票据字段（商户/金额/日期）
2. 大写金额识别（专门 prompt）
3. 用户补银行流水 → :8766 → 按 日期 + 金额 + 商家 三维匹配
4. 内置税率表 → 自动算 增值税/附加税/企业所得税
```

## 五、前端关键模块

| 模块 | 文件 | 作用 |
|------|------|------|
| 主 SPA | `snapextract_v3.html` | 单文件 ~18k 行：UI / 路由 / 4 场景管线 |

## 七、对比传统 OCR

| 维度 | Tesseract / 普通 OCR | 答于言 |
|------|--------------------|--------|
| 文字识别 | ✔（印刷体好，手写差） | ✔ Qwen 视觉直读，手写体远强于 OCR |
| 结构化抽取 | ✘ 只输出纯文本 | ✔ 按场景输出字段化 JSON |
| 多模态理解 | ✘ | ✔ 公式 LaTeX / 图表说明 |
| 业务判断 | ✘ | ✔ 论文价值 / 简历评分 / 合同风险 / 对账核对 |
| 数据安全 | ✔ 本地 | ✔ 本地（NPU 加持，速度可用） |
